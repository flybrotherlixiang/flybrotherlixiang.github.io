General List: 移动端动态化模板引擎

General List 是百姓网客户端内部采用的一套轻量级的动态化渲染框架。它基于客户端和服务端预先定义好的一组模板来动态地下发结构化的客户端界面展示样式，组装上路由协议（Router）后更能轻松支持跳转至任意界面的能力，做到灵活地展示 + 跳转。

前言

「App 动态化」一直以来都是移动端开发的热门话题之一。 动态化方案的技术选型也层出不穷，无论是早期迁移成本低的 Hybrid 混合开发，还是后来日益成熟的基于 Web 标准的 React Native 等框架，都是为了能更好地服务于快速更迭的业务需求。

这篇文章将带来「General List」这一轻量级动态化框架的实现思路，看看它是如何应对某些业务场景下的动态化需求的。

起因

App 因其顺畅的操作体验和可离线使用等特性成为智能手机上最佳的产品展现形态。然而相对地，版本更新不可控的局限性也制约了产品上新功能的快速迭代、落地：

  • 应用市场上新流程冗杂，如果是 iOS App 还要虔诚地祈祷审核不出问题 🙏
  • 线上存在着多个历史版本：
    • 服务端接口的部分模块需要针对历史版本作兼容处理
    • 覆盖安装的情况下，客户端新版本的本地数据迁移
  • 有些用户就是死活不更新咬我呀

于是产品经理们对 App 动态性的迫切渴望化为了手中一条条炽热发烫的皮鞭，抽打着开发者们持续为客户端加持更强的远程可配置性…

目标

为了让产品的新特性能更快地触及到每一个用户，机制上对这套动态化方案的效果有以下两点要求：

  • 不需要发布新版本，远程控制 App 上特定页面的展示效果
  • 不需要发布新版本，远程控制各个视图区块被用户点击后的处理行为

技术选型

我们考虑过以下几种技术方案：

1. React Native、Weex 等多平台开发框架

比较理想的方案，在不牺牲性能和流畅性的同时还能保证多端的开发体验基本一致；有网络时就可以通过下发 js bundle 来改变客户端界面的展示和内部逻辑，让客户端的更新更快地体现出来。

然而在 2015 年我们考虑实现动态化方案的时候 React Native 才刚起步没迭代几个版本（七月份刚到 v0.7.0 左右），社区生态都远不及现在成熟，而 Weex 2016 年才正式发布，考虑到接入的风险成本我们最终没有选择拥抱该技术。

2. 各平台的热修复方案

以 iOS 上的 JSPatch 为例，借助 Objective-C 的 runtime 机制桥接了 Native 和远程下发的 JS 代码，从而达到热更新的效果。热修复框架很强大，理论上可以做到不发版动态替换和增添客户端的表现和处理能力，但当时各家的热修复框架还没有稳定到可以承载大量业务代码的地步，存在一定的出错风险，另一方面我们也不希望把修复和需求开发掺和在一起让项目流程变得相对混乱。

3. Hybrid

把一部分原生的功能改为内嵌的 HTML5 来开发，借由 Web 的特性在不发布 Native 新版的情况下动态更新内容。同样是多端基本一致的开发体验，操作的流畅性要稍弱于 Native 实现的版本。

混合开发的实时性可以满足上述的动态化需求，跨平台的特点也缩短了开发时间和成本。但是在一些设备和系统上的适配需要额外的精力去做尝试，此外还要仿照 Native App 的 UI 去实现一部分组件。在客户端技术人员充足且 Code Base 几乎都是 Native 代码的前提条件下，投入产出比可能不会太高。

4. Native 实现

事实上上述方案的效果已经大大超出了既定目标，因为它们能全量地更新客户端所有的代码实现，包括网络层、存储层或任意需要修改的逻辑。如果只对界面的展示与跳转有远程控制的要求，能否通过纯 Native 的技术手段来实现呢？于是我们设计了轻量级的动态化框架 —— General List，它基于客户端和服务端预先定义好的一组模板来动态地下发结构化的客户端界面展示样式，组装上路由协议（Router）后更能轻松支持跳转至任意界面的能力，做到灵活地展示 + 跳转。

客户端架构

基于 General List 的客户端渲染流程并不复杂，如上图所示：

  1. App 在用户进入相应界面后先发起请求向服务端询问所需要的 JSON 原始数据 ，通常是个元素类型为字典的数组。
  2. 接收成功后客户端逐条解析数组中的元素，根据字典中最核心的 style 字段确定该字典所要输出的对应的本地模型实例（Model），校验数据有效性的同时提高了后续读取的便利性。
  3. 每个版本的客户端中都装载了一组预定义的样式模板，同时每个 Model 都指定了一个与之对应的模板，这时候拿着这一串 Models 填充进它们对应的模板，得到最终用户可见的视图（ Views ）。

这个所谓”模板”究竟是指什么？

其实就是业务无关的朴素的视图组件，如你写的一个 iOS 中 UIView 的子类。

它们各自拥有独特的外观，但核心元素的内容是外部可以自由装配的。

如下左图定义了一种上图、下文字的模板（下称模板 A），图片和文字的内容是外部注入的；而下右图定义了一种上视频、下左头像、下右偏下文字、下右偏上文字的模板（下称模板 B），它们都是传统意义上纯粹的展示层。

如何确定每个模板的最终布局？

我们知道最终绘制在屏幕上的界面背后是由一棵视图树来支撑的，原则上每个视图只定义它内部的布局，其自身的位置（有些时候包括尺寸）需要由父级容器来指定。为了控制模板的动态展示，我们通过下发树形结构的模板元数据来定制布局。

响应点击事件

通过合理地调配数据，我们已经能基于模板库组合出大部分场景下的页面展示了 🎉。

另一方面，当用户点击某一区块后程序如何响应这一点击事件？

我们希望响应的部分是和展示无关的。

比如上面的模板 B，业务方可能想让有的区块点击后跳转到播放该视频的详情页，而另一些区块在展示上和其他人一样，但本质上是伪装成这个样式的第三方广告，那么点击后就可能是跳转到某个 WebView 里去了。

能否远程控制客户端各区块的点击事件响应？

在我们的项目中，General List 依托路由模块（ Router ），统一通过自定义的 URL 解析来赋予它灵活跳转的能力。关于 App Router 的实现业界已有很多成熟的设计方案和思路这边就不再赘述了。简单地说，每个 App 可以定义一套自己的路由协议（如百姓网的 bxapp://xxx)，在程序启动时注册好 URL 和对应组件的关系，之后就能每次通过解析输入 URL 中的 target、action 及相关参数来决定要执行的逻辑了。

局限性 & 后续优化

General List 驱动的界面渲染在百姓网客户端中应用于各类对动态化有需求的场景中，包括常规的列表和变动频繁的运营页等。它的局限性也比较明显：服务端只能下发客户端该版本预定义过的模板用于展示，这就对模板库的多样性提出一定要求。为了进一步深化模板的变换能力，可以考虑结合 React Native，做到”真正”地下发并在客户端创造自定义模板，

另外，服务端 API 接口需要依据版本清晰地界定出客户端支持的模板，一定程度上提高了服务端的维护成本。理想情况下可以通过搭建一个模板配置后台来让整个流程运转地更顺畅，运营同学或开发者可以在后台方便地编辑或查看某个模板的名字、相关的字段及类型、所属版本、对应的渲染结果等，客户端和服务端再通过一些数据同步机制来避免下发样式无效的问题，从而提高系统的稳定性。

结语

虽然目前各大厂在实现动态性时会以 Hybrid 和 React Native 为主，但对于一些相对轻量的业务来说考虑用 General List 这种思路来驱动展示也能很好地适应需求，希望对大家有所启发~
